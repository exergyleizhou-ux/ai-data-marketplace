# Step 3 - MVP 技术栈最终建议 + 关键技术验证点

**日期**：2026-05-31  
**依据**：P3 用户故事 + 已确认决策（中文文本 + 代码优先、资金托管 10% 佣金、基础质量检测）

---

## 1. 推荐技术栈（MVP 阶段）

### 强烈推荐方案（平衡开发效率 + 性能 + 国内生态）

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| **后端语言/框架** | **Go + Gin / Fiber**（推荐） | 并发性能强、大文件处理友好、编译快、部署简单。国内大模型公司后端大量使用 Go |
| **前端** | **Next.js 14 (App Router) + TypeScript + Tailwind** | 开发效率高，全栈可复用，SEO 对数据集列表友好 |
| **数据库** | **PostgreSQL 16**（主库） + **Redis** | 关系型数据强，事务可靠。Redis 做缓存 + 分布式锁 |
| **对象存储** | **阿里云 OSS** 或 **腾讯云 COS**（推荐国内） | 国内上传下载速度快，价格合理，支持分片 + 预签名 URL |
| **任务队列** | **Asynq**（基于 Redis）或 **RabbitMQ** | Asynq 足够轻量，适合上传后处理、质量检测 |
| **搜索** | **PostgreSQL + pg_trgm**（初期） → 后期 Elasticsearch | P3 数据量不大，先用数据库模糊搜索即可 |
| **支付集成** | **微信支付 + 支付宝官方 SDK** + **Stripe**（国际） | 必须支持国内主流 + 国际 |
| **容器 & 部署** | **Docker + Docker Compose**（初期）<br>后期上 Kubernetes 或阿里云 ACK | MVP 阶段强烈建议先用云服务，降低运维 |
| **监控 & 日志** | **Prometheus + Grafana** + **阿里云 SLS** 或 **Sentry** | 必须有 |
| **鉴权** | **JWT + Refresh Token** + **Casbin**（RBAC） | 简单可靠 |

**备选方案对比**：

| 方案 | 优点 | 缺点 | 推荐指数 |
|------|------|------|----------|
| Go + Gin | 性能、并发、部署 | 生态不如 Node 丰富 | ★★★★★ |
| Node.js + NestJS | 开发快，全栈 JS | 大文件处理相对弱 | ★★★☆☆ |
| Python + FastAPI | 适合 AI 团队内部 | 并发一般，部署稍重 | ★★☆☆☆ |

**最终建议**：**Go 后端 + Next.js 前端** 是当前最优平衡。

---

## 2. 关键基础设施选型建议

- **对象存储**：强烈建议用**国内云厂商**（阿里云/腾讯云），不要用 AWS S3（国内上传体验差）。
- **云服务商**：阿里云 或 腾讯云（二选一即可）。初期全部托管在上面。
- **域名 & HTTPS**：必须用正规域名 + 阿里云/腾讯云 SSL 证书。
- **CDN**：数据集下载量上来后必须加 CDN（尤其是代码仓库类数据）。

## 3. P3 必须提前验证的 6 个技术风险点

这些是 P3 阶段最容易翻车的点，**建议在正式开发前 1-2 周做技术验证（Spike）**：

### 验证点 1：大文件分片上传 + 断点续传（最高优先）
- 目标：支持 5GB+ 文件稳定上传
- 验证内容：
  - 前端分片 + 断点续传逻辑
  - 后端合并逻辑 + 完整性校验
  - 国内弱网环境下的表现
- 建议工具：阿里云 OSS 分片上传 SDK

### 验证点 2：资金托管支付闭环（最高风险）
- 目标：实现“买家支付 → 平台托管 → 买家确认 → 自动结算”完整流程
- 验证内容：
  - 微信/支付宝是否支持资金池/担保交易模式（或用“分账”功能模拟）
  - 退款、纠纷场景的资金处理
  - 与平台账户体系的打通
- **这是 P3 最大商业风险点**，必须提前确认可行性

### 验证点 3：基础自动化质量检测 Pipeline
- 目标：上传后自动跑通格式检查 + 简单统计 + 去重
- 验证内容：
  - 大文本/代码文件的流式处理（不能全加载进内存）
  - 简单去重算法（MinHash / SimHash）
  - 任务队列 + 结果回写数据库

### 验证点 4：安全的数据交付机制
- 验证内容：
  - 预签名 URL 的时效控制
  - 下载时注入买家信息（水印）
  - 防盗链 + Referer 限制

### 验证点 5：实名认证对接
- 验证内容：
  - 对接第三方实名认证服务（阿里云实人认证、腾讯云、或公安接口）
  - 企业营业执照 OCR + 核验

### 验证点 6：多支付渠道统一抽象
- 目标：后续能轻松增加新支付方式
- 建议：做一层 Payment Gateway 抽象

## 4. 推荐的 MVP 开发与部署节奏

**阶段 A（验证期，2-4 周）**
- 完成上面 6 个验证点中的前 3 个（尤其是支付托管和分片上传）
- 决定最终技术栈

**阶段 B（核心开发）**
- 按用户故事优先级顺序开发（见 Step2 文档）

**阶段 C（部署上线）**
- 初期推荐：**阿里云 / 腾讯云** 一条龙托管（ECS + RDS + OSS + SLS）
- 避免自己搭 Kubernetes，成本和复杂度太高

---

## 5. 团队与人力建议（P3 阶段）

假设 4-6 人团队：

- 1 后端（Go） - 主力
- 1 前端（Next.js）
- 1 全栈 / 后端（支付 + 质量检测）
- 1 产品 + 运营（负责审核 + 内容）
- 1 兼职/外包做支付集成与资金流验证（最难）

**最需要外包/专家支持的部分**：支付托管流程 + 资金合规。

---

**结论**：
- 技术栈推荐：**Go + Next.js + PostgreSQL + 阿里云/腾讯云**
- 最该先做的两个验证：**分片上传** 和 **资金托管支付闭环**
