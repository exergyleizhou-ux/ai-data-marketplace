# 资金分账合规与集成方案

> **定位**:这是一份**工程接口契约 + 合规落地路线图**,给"接入真实支付分账"这条线用。
> **它不是法律意见**。任何真实资金通路上线前,必须经**执业律师/合规顾问**审核资金流向、并与**持牌支付机构/银行**签约。代码默认**不启用**任何真实通道(`server.go` 在未配置时回退到 sandbox mock)。

---

## 0. 一句话

平台**永远不碰钱**:买家付的钱由**持牌方**(持牌支付机构或银行)托管,平台只通过 API**下达分账指令**,持牌方把卖家应得部分**直接结算给卖家**、平台只收技术服务费/佣金。这样才不构成"二清"。

## 1. 红线:什么是"二清",为什么必须避开

- **二清(二次清算)** = 没有支付牌照的主体,先把多个收款人的钱归集到自己账户,再由自己拆分转给各收款人。
- 在中国,无证从事支付结算业务,涉嫌**非法经营罪**(刑事),是本项目的**生死线**(见 [`设计与实施文档-v2.0.md`](设计与实施文档-v2.0.md) §2.1)。
- 典型违规姿势(**禁止**):买家付款 → 进**平台对公账户/平台在通道的单一商户号** → 平台再转账给卖家。
- 合规姿势(**唯一可接受**):买家付款 → **持牌方托管** → 持牌方按平台指令**分账直达卖家**与平台佣金账户;平台账户里只会出现**自己的佣金**,从不出现卖家的本金。

## 2. 合规三原则(任何通道都必须满足)

1. **资金不落平台自有账户**:卖家本金永远不进平台可支配的账户。
2. **平台只下指令**:平台调用持牌方的分账 API,持牌方执行清算。
3. **可追责留痕**:每笔订单的收款、分账、退款都有持牌方流水号 + 平台侧 `audit_logs`(只追加)对账。

## 3. 资金流向图

```mermaid
sequenceDiagram
    participant B as 买家
    participant P as 平台(本系统)
    participant L as 持牌方(支付机构/银行)
    participant S as 卖家

    B->>P: 下单
    P->>L: CreatePayment(orderID, amount, 分账规则)
    L-->>B: 收银台/二维码
    B->>L: 付款(资金进入持牌方托管,不经平台)
    L-->>P: 异步回调 VerifyCallback(已支付)
    P->>P: 订单→paid;触发交付
    Note over P,L: 买家确认收货后
    P->>L: ExecuteSplit(orderID, sellerRef, 卖家90%, 平台10%)
    L->>S: 直接结算卖家本金(90%)
    L->>P: 结算平台佣金(10%)
    Note over P,L: 纠纷退款时
    P->>L: Refund(channelTxn, splitTxn, amount)
    L->>S: 冲正已分账给卖家的部分(如已分账)
    L->>B: 退款给买家
```

纯文本版:`买家 → 持牌方托管(钱不经平台) → [回调]平台标记已付 → [确认收货]平台下分账指令 → 持牌方直接结给卖家90%、平台收佣10% → [退款]平台下退款指令 → 持牌方冲正+退买家`。

## 4. 通道选型对比(三选一)

| 方案 | 谁扛二清合规 | 资质门槛 | 落地速度 | 说明 |
|------|------------|---------|---------|------|
| **A. 持牌聚合/分账服务商**(连连、易宝、汇付天下、富友、Ping++ 等) | 服务商(持牌) | 营业执照 + 对公户 + 类目审核 | **最快** | 服务商提供账户体系/分账 API,平台对接即可;最推荐先谈 |
| **B. 银行电子账簿/存管分账** | 银行 | 较高,需对公合作 | 中 | 资金银行存管,合规最稳,接入周期较长 |
| **C. 微信支付/支付宝官方分账** | 微信/支付宝(持牌) | **最高**:特约商户 + 分账权限申请 + 类目/担保交易限制 | 慢 | 直连,但资质与类目门槛高,数据交易类目能否开通需确认 |

**建议**:先谈 A(服务商),并行让法务评估 B/C;最终以法务对**资金流向**出具的合规意见为准。

## 5. 平台 ↔ 持牌方 接口契约(已在代码里就位)

真实通道**只需实现现有接口**,无需改动业务逻辑。接口定义见 [`backend/internal/modules/payment/provider.go`](../backend/internal/modules/payment/provider.go),Stripe 实现 [`stripe.go`](../backend/internal/modules/payment/stripe.go) 就是范例。

| 接口方法 | 持牌方对应动作 | 备注 |
|---------|--------------|------|
| `PaymentProvider.CreatePayment(orderID, amountCents)` | 发起收款单,返回收银台/二维码 URL + 通道流水号 | 资金进**托管**,非平台账户 |
| `PaymentProvider.VerifyCallback(payload, signature)` | 验签 + 解析异步通知 → `{orderID, channelTxnID, paid}` | 客户端**绝不**自标支付,只认回调 |
| `SplitProvider.ExecuteSplit(ctx, orderID, sellerRef, sellerAmount, platformFee)` | 按指令分账:卖家本金直达卖家,平台收佣 | `sellerRef` = 持牌方侧卖家收款标识 |
| `RefundProvider.Refund(ctx, channelTxn, splitTxn, amount)` | 冲正已分账(如有)+ 退买家 | `splitTxn==""` 表示尚未分账,无需冲正 |

幂等/可靠性已由现有 H3 outbox + advisory lock + Stripe `IdempotencyKey` 模式覆盖;新通道实现 `ExecuteSplit` 时同样要传一个稳定幂等键(如 `settle:<orderID>`)。

### 5.1 `sellerRef` 与卖家收款账户绑定

- `sellerRef` 是**持牌方侧**的卖家收款标识(子商户号 / 分账接收方 ID / 绑定的银行卡)。
- 复用现有 **`payout_accounts`** 表(auth 模块拥有,H1 落地):`(user_id, channel, account_ref)`,`channel` 取 `"wechat"` / `"alipay"` / `"<服务商名>"`。
- 绑定时机:卖家完成 **KYC**(实名,已就位)后,引导其在持牌方开通/绑定收款,把返回的标识 `SavePayoutAccount(userID, channel, accountRef)` 持久化。
- 分账时 `payment` 模块通过 `PayoutAccountStore` 适配器读取 `sellerRef`(与 Stripe 路径完全一致)。

## 6. 代码接入点(等签约+法务后)

1. 新建 `backend/internal/modules/payment/<channel>.go`,实现 `PaymentProvider + SplitProvider + RefundProvider`(照 `stripe.go`)。
2. 在 [`server.go`](../backend/internal/server/server.go) 的 provider 选择块(约 235 行)加 `case "wechat"/"alipay"/...`,仅当**生产密钥已配置**时启用,否则维持回退 mock。
3. `Channel()` 返回通道名;`payout_accounts.channel` 用同名。**迁移无需改**(channel 是字符串,000004 的 CHECK 如需新增取值再加一条迁移)。
4. 新增配置项(env,放 KMS,绝不进仓库):`<CHANNEL>_MERCHANT_ID` / `<CHANNEL>_API_KEY` / `<CHANNEL>_API_SECRET` / 回调验签证书等。

## 7. 上线前 checklist(逐项打勾才可开真钱)

- [ ] 公司资质齐备:营业执照、对公账户、(如需)经营类目/行业资质。
- [ ] **法务出具资金流向合规意见**:确认不构成二清(钱不落平台账户)。
- [ ] 与持牌方签约,拿到生产商户号/密钥/分账方资质要求。
- [ ] 卖家侧:KYC + 持牌方收款绑定流程跑通,`payout_accounts` 落库。
- [ ] 生产密钥进云端密钥管理(KMS/Secrets),`git grep` 确认仓库零密钥。
- [ ] 回调验签、幂等、对账(settlements 唯一键 + outbox + audit_logs)全链路灰度验证。
- [ ] 退款/分账冲正路径(H2)在 sandbox 验证通过。
- [ ] 灰度:小额真单端到端 → 放量。

## 8. 边界声明

- 本文档与代码仅提供**技术接入能力**;是否、何时、以何种结构接入真实资金,由**法务 + 持牌方**决定。
- 在满足第 7 节全部前置前,系统**默认且必须**停留在 sandbox(`PAYMENT_PROVIDER=mock`)或已验证的 Stripe 测试模式。
- 微信/支付宝**境内真实分账**仍是 Spike-2 + 法务事项,不得绕过。
