# Oasis — A Verifiable-Evidence Layer for High-Stakes Computation

# 可验证证据层 · 高风险计算的可信底座

> **一句话 / TL;DR**
> Run an audited algorithm against data you can't see, inside a network-severed
> sandbox, and walk away with a certificate that *anyone* can re-verify by
> re-hashing the result — binding **this exact output** to **this audited
> algorithm image** and **this dataset**. The honest claim is **provenance**, not
> privacy: the certificate proves *what produced what*, and the system is candid
> about everything it does **not** prove.

This is a technical showcase of the compute-to-data (C2D) core inside
[Oasis](https://github.com/exergyleizhou-ux/ai-data-marketplace). It is written to
be read by a skeptic. Every claim below is either backed by code + tests in this
repo, by a live certificate you can re-hash yourself, or is explicitly marked as a
**gap**.

---

## 1. The thesis: provenance over privacy

A lot of "privacy-preserving compute" pitches over-claim. This project deliberately
makes the *narrower, defensible* claim:

- **What it is:** a verifiable-evidence layer. A computation runs in a hardened
  sandbox; the result is bound, by content hash, to the digest-pinned algorithm
  image and the source dataset; the binding is published so anyone can check it.
- **What it is NOT (stated up front):** it does not, by itself, prove the operator
  cannot see the data (that needs a TEE — see §6), nor that the algorithm's
  *conclusion* is correct, nor that a malicious algorithm couldn't leak a little
  within bounds (see §4). Those limits are disclosed in the
  [threat-model doc](威胁模型与保证-C2D可验证证据层.md) and on the live
  [`/c2d/honesty`](https://github.com/exergyleizhou-ux/ai-data-marketplace) page.

The honest through-line is **verifiable, reproducible provenance** for computations
where you can't (or shouldn't) move the raw data around.

## 2. How it works

```
buyer buys "run algorithm A on dataset D"
        │
        ▼
operator stages D read-only  ──►  hardened sandbox  ──►  single output object
(before the network is cut)        docker run \             │
                                     --network=none \        │ the ONLY way out
                                     --read-only \           │
                                     --user 65534 \          ▼
                                     --cap-drop=ALL      output gate (§4)
                                                             │
                                                             ▼
                       result stored ──► certificate: sha256(output)
                                          ⟵ produced by ⟶ image@sha256:… + dataset
                                                             │
                                                             ▼
                                    public GET /verify/<cert>  +  re-hash yourself
```

- **Network-severed sandbox** (`backend/internal/modules/compute/runner_docker.go`):
  `--network=none --read-only --user 65534:65534 --cap-drop=ALL
  --security-opt=no-new-privileges`, resource-capped, dataset mounted `ro`. The
  output object is the *only* exfiltration channel.
- **Digest-pinned algorithms:** a trusted algorithm is referenced as
  `image@sha256:…`, never a mutable tag — the audited code can't be silently
  swapped.
- **Aggregates only:** every shipped algorithm returns summary statistics / fit
  parameters, never raw rows.
- **Certificate + public verify:** the output's SHA-256 is bound to the image
  digest + dataset and registered for an unauthenticated `GET /verify/:cert_id`.

## 3. What the certificate guarantees — and what it doesn't

| Guarantees | Does NOT guarantee |
|---|---|
| **Provenance** — this output came from this digest-pinned algorithm on this dataset | **Correctness** — a buggy/biased algorithm still yields a valid certificate |
| **Integrity** — the output wasn't altered; the image wasn't swapped | **Safety** — bounded exfil *within* the gate is still possible (§4) |
| **Reproducibility** — method + data + result is a re-runnable, tamper-evident record | **Operator-invisibility** — under the default L1 the operator can see staged data (§6) |

Full four-adversary analysis (curious buyer / malicious buyer / malicious
**algorithm author** / compromised operator): see
[`docs/威胁模型与保证-C2D可验证证据层.md`](威胁模型与保证-C2D可验证证据层.md).

## 4. The engineering that makes it trustworthy

The difference between a demo and an evidence layer is the adversarial details.
Each of these is test-driven (`*_test.go`) and merged on `main`.

- **Anti-malicious-algorithm output gate** (`output_gate.go`). The sandbox cuts the
  network, so a *malicious algorithm author* could try to steganographically pack
  raw rows into a "aggregate". The gate enforces a structured shape (a JSON object,
  or a zip of `*.json`) and bounds the information content — total string bytes,
  numeric-value count, nesting depth, and a Shannon-entropy check on long strings —
  fail-closed. It honestly only *bounds* exfil, it doesn't zero it; the residual is
  disclosed.
- **Real Intel TDX attestation verification** (`runner_tee_dcap.go`). Rather than
  hand-rolling attestation crypto (easy to get subtly wrong), it wraps Google's
  audited `go-tdx-guest` to verify a **real Intel production TDX quote** offline:
  ECDSA signature + PCK certificate chain to Intel's root + a measurement
  allowlist. Unit-tested against Intel's published sample quote. (The *hardware*
  half — emitting a live quote — is gated; see §6.)
- **L2 fail-closed enforcement** (`worker.go`). An L2 (confidential-compute) offer
  must produce an attestation report; a TEE runner that returns none fails the job
  closed rather than releasing output as if it were attested.
- **Differential privacy** (`dp.go`): a central Laplace mechanism with an atomic
  budget ledger; the epsilon is platform-injected and the buyer can't disable it.
- **TDD throughout.** Every feature above landed RED→GREEN with a failing test
  first; the compute module alone carries dozens of real-Postgres integration
  tests.

## 5. Demonstrated on REAL public data

To avoid the "it only works on synthetic demo data" critique, here are two live
certificates produced by running the **real, openly-licensed UCI Wine Quality
(red)** dataset (1,599 rows) through the production docker sandbox. Both are
re-hash-verifiable by anyone.

| Analysis | Result (real) | Certificate | Re-hash |
|---|---|---|---|
| **Integrity screen** (PaperGuard, 8 detectors) | 1,599 rows · 5/8 detectors flagged · 46 findings · verdict `anomalies_flagged` | `VO-0C300E9FAED2` | `sha256:e5db8e08…` ✓ matches |
| **Causal ATE** (alcohol → quality, adj. pH/sulphates/density) | OLS ATE **0.365** (95% CI 0.33–0.40, p≈0); cross-fitted DML **0.321** (CI 0.27–0.37) | `VO-2B9CCA235122` | downloaded output re-hashes to the cert ✓ |

Two honest observations that make these *more* credible, not less:

1. The integrity screen flags **anomalies, not fraud.** A famous, legitimate
   published dataset still trips digit-distribution detectors (natural rounding /
   measurement heaping). The certificate records the screen's *operating
   characteristics*, it does not adjudicate fraud — exactly as the
   [PaperGuard validation](验证-PaperGuard筛查统计操作特性.md) documents.
2. The causal estimate (alcohol raises rated quality by ~0.32–0.37 points per unit,
   adjusted) is a *sensible, real* result — and the certificate proves it was
   produced by the audited estimator on this dataset, **without claiming the causal
   assumptions hold**. Provenance, not correctness.

Reproduce either independently: re-hash the downloaded result, check
`GET /verify/<cert>`, or re-run the digest-pinned image on your own data — see
[`docs/复现-独立验证C2D结果.md`](复现-独立验证C2D结果.md).

## 6. Honest gaps — what needs external input, not more code

These are stated plainly because over-claiming is the failure mode this project is
built to avoid. Two of three are cheaply closable; one should wait.

| Gap | Status | Closes with |
|---|---|---|
| **Operator can see L1 data** | disclosed; L2 **verification half is real** (§4) | a real TDX node emitting a live quote (a few \$ of cloud confidential-VM time) |
| **Real, sensitive, multi-party data** | demos use real *public* data (above) + model-grounded synthetic, both labelled | a genuine data owner running their own private data through the pipeline |
| **Compliant payment / settlement** | sandbox / test-mode only | a licensed payment provider — deliberately deferred until there's real revenue |

There is no synthetic "10th algorithm" or fabricated privacy feature here. The
system knows what it is.

## 7. Verify any of this yourself

- **Re-hash a result:** download a job's output, `shasum -a 256`, compare to the
  certificate's `output_sha256`.
- **Public verify (no login):** `GET /verify/VO-0C300E9FAED2`.
- **Re-run the algorithm:** pull the digest-pinned image and run it under the same
  `--network=none --read-only --user 65534` posture on your own data.
- **Read the code:** the sandbox is `runner_docker.go`; the output gate is
  `output_gate.go`; the TDX verifier is `runner_tee_dcap.go`; the threat model is
  [`docs/威胁模型与保证-C2D可验证证据层.md`](威胁模型与保证-C2D可验证证据层.md).

*Everything above is on `main`, CI-green, and (for the live certs) re-verifiable as
of this writing.*
