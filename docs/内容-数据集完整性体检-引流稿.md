We built **Oasis Verify** — an API that screens a dataset for statistical
integrity anomalies and hands back a **re-hashable certificate** binding the
result to a digest-pinned, audited algorithm. To dogfood it, I pointed it at three
datasets every ML person has touched.

All three are legitimate, decades-old, widely-cited UCI datasets. **All three trip
integrity detectors.** Here's the honest punchline up front: that does **not** mean
they're fraudulent. It means the screen is doing its job — surfacing
digit-distribution and rounding patterns that are *normal* in real measured data.
A flag is a "look here," not a verdict.

Each ran in a `--network=none --read-only` sandbox; only the aggregate report
leaves; the raw rows are never exposed. You can re-hash the output and check it
against the cert, or look it up at `/verify/<cert>` without an account.

## What the detectors actually caught

- **Terminal-digit / last-digit heaping** — measured quantities cluster on round
  values (think pH read to one decimal, integer counts). Real and expected.
- **Benford-ish first-digit deviations** — small bounded features (petal widths,
  oxide percentages) don't follow Benford, and shouldn't. The detector flags the
  deviation; a human reads the context.
- **Decimal over-consistency** — fixed instrument precision looks "too clean."

None of this is fraud. The point of an integrity screen isn't to accuse — it's to
give you a **fast, shareable, third-party-verifiable** signal of *where to look*
before you train on, buy, or publish a dataset.

## Why a certificate (vs. just running checks locally)

You can run these checks yourself. What you can't easily do is **prove** you ran
them, on exactly this data, with exactly this method, in a way someone else can
verify without trusting you. That's the certificate: output fingerprint ⟵ produced
by ⟶ audited algorithm (pinned image digest) + source dataset. Re-runnable,
tamper-evident.
